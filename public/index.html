    📖 Anleitung — AlpineBits Sample Manager
    🏔️

    AlpineBits Sample Manager

    Dieses Dashboard verwaltet XML-Testdaten für den AlpineBits HotelData Standard — das offizielle Protokoll für den Datenaustausch zwischen Hotelsoftware-Systemen in den Alpen.

    🎯 Was ist AlpineBits?

    AlpineBits ist ein offener Standard für die Kommunikation zwischen Hotel-Property-Management-Systemen (PMS), Channel-Managern, Buchungsplattformen und OTAs. Er basiert auf dem OTA-Schema (OpenTravel Alliance) und definiert XML-Nachrichten für:

    🤝Handshake

    Verbindungsaufbau & Capability-Austausch

    🛏️FreeRooms

    Verfügbarkeits-Updates: freie/belegte Zimmer

    💶BaseRates

    Basis-Preisabfragen zwischen Systemen

    📋RatePlans

    Preisplan-Notifikationen & Sonderangebote

    👤GuestRequests

    Buchungen, Reservierungen, Stornierungen

    🏨Inventory

    Stammdaten: Zimmertypen, Ausstattung

    🎿ActivityData

    Aktivitäten & Events rund ums Hotel

    📊 Offizielle Samples vs. Unsere Testsammlung

    🏛️ Offizielle Samples
    • ~54 XMLs aus dem AlpineBits GitLab
    • Live geladen — immer aktuell
    • Hauptsächlich Happy-Path Beispiele
    • Kaum Negativ- oder Invalidierungstests
    📦 Erweiterte Testsammlung
    • 141 XMLs — mehr als doppelt so viele
    • Systematische NEG-Tests: Was Server ablehnen müssen
    • Systematische INV-Tests: XSD-Fehler die erkannt werden müssen
    • Edge Cases: Delta/CompleteSet, Capabilities, Grenzwerte, Saisonen

    🆕 Latest

    Jede XSD-Datei wird alle 12 Stunden gegen das offizielle GitLab-Repository geprüft. Der Header zeigt den aktuellen Status. Bei Änderungen: ⚠️ XSD geändert — dann sollte ein Retest durchgeführt werden.

    💡 Wichtig: Jedes Zertifikat gilt für eine bestimmte Spec-Version. Wenn 2024-10 erscheint, bleibt ein 2022-10-Zertifikat gültig — es ist nur noch kein 2024-10-Zertifikat.

    🏷️ XML-Typen Legende

    RQ Request — wird vom Client an den Server gesendet; muss XSD-valide sein
    RS Response — Serverantwort; muss XSD-valide sein
    NEG Negativfall — XSD-valide, aber Server MUSS aufgrund von Geschäftslogik ablehnen
    INV Invalide — absichtlich fehlerhaftes XML; XMLlint MUSS Fehler melden (Validierungslauf: ✅ wenn fehlschlägt)
    contrib Community — von der Community beigesteuerte Beispiele
    ⚠️ Hinweis zu INV-Tests & Spec-Versionen: Manche INV-Dateien testen Regeln, die erst ab einer bestimmten Spec-Version im XSD erzwungen werden. Beispiel: INV-02-missing-hotelcode schlägt korrekt nur gegen 2024-10 fehl (HotelCode-Fix). Das ist kein Fehler, sondern ein wertvoller Hinweis auf XSD-Schwächen in älteren Versionen.

    🤖 KI-Generator — Details

    1
    Standard-Nutzung (20 Calls/Tag)

    Ohne eigenen Key: kostenlos nutzbar, max. 20 KI-Generierungen pro Tag. Reset automatisch nach 24h.

    2
    Eigener Anthropic API-Key (unbegrenzt)

    Unter 🔑 Eigenen Anthropic API-Key verwenden den eigenen Key eintragen (sk-ant-...). Der Key verlässt nie den Browser — er wird direkt an die Anthropic API gesendet, ohne Umweg über den Server.

    3
    Spec-Version für Generierung wählen

    Im Validierungsschritt nach der Generierung kann die Ziel-Spec-Version gewählt werden. Empfehlung: gegen 2022-10 (Default) und 2024-10 (Latest) testen.

    💡 Tipp Validierungslauf: Erst mit 2022-10 laufen lassen (Baseline), dann mit 2024-10 vergleichen. Unterschiede zeigen wo die neue Spec strenger wurde.

    📌 Alle Validierungen nutzen xmllint mit dem offiziellen alpinebits.xsd — dasselbe Tool wie in der offiziellen AlpineBits Test-Pipeline.
